# Vibe - ビジネスコンセプト

## ビジョン・思想
- 非エンジニアの経営者・事業者が「アイデアと言葉」だけで業務システムを手に入れる世界をつくる。
- GitHub アカウントもリポジトリも不要。ローカル開発も不要。UI は「左チャット + 右プレビュー（Claude Artifact風）」の 2-pane 体験で完結。
- LP を作る AI サイトビルダーではなく、Next.js + DB + 認証 + EC + 管理画面まで含むフルスタック Web アプリを自動生成する「AI アプリ自動生成インフラ」。
- 生成の中心はカスタムプロンプト。設計思想・ルール・生成フロー・デプロイロジックを一枚の標準仕様として保持し、これこそが Vibe の IP。

## 解決したい経営課題と提供価値
- 課題：エンジニア採用や外注のコストが高く、要件整理も難しい。ノーコードも学習・運用コストが大きく、LP だけでは売上や業務効率化に直結しない。
- 解決：ビジネスゴールを 3 点ほど確認するだけで、AI が要件抽出→設計→実装→検証→デプロイまで代行。経営者は「作りたいものをざっくり伝える」だけで、稼働するフルスタックアプリを受け取れる。
- 価値：
  - preview デプロイが自動で生成され右ペインに即表示。気に入ればコードを買い切り、または SaaS として運用。
  - DB スキーマ、認証、決済（Stripe）、管理画面、画像アップロードなど業務運用に必要な要素を初期から同梱。
  - GitHub 操作不要。必要なら本番用 Workflow もプロンプトから自動生成し、オーナーはコード所有権を得られる。

## なぜ AI サイトビルダーではないのか
- 目的は LP ではなく「業務が回る完成アプリ」の提供。API、Server Actions、DB、決済、認証、管理画面まで生成し、運用に耐える形で引き渡す。
- 生成物は断片的なコードではなく、Next.js プロジェクト全体とデプロイフロー。プレビューと本番を見据えた CI/CD（GitHub Actions も自動生成）まで含む。

## Kiro / CC-SDD との明確な差別化
- Kiro / CC-SDD：エンジニア向け、既存リポジトリ前提、GitHub 必須、コード編集が前提の「エンジニア支援ツール」。
- Vibe：非エンジニア向け、リポジトリ不要、要件を話すだけでアプリをフル生成し自動デプロイする「完成アプリ自動生成」。カテゴリが異なるため直接競合しない。

## カスタムプロンプトこそがプロダクトの本質
- 一枚のカスタムプロンプトに、ビジネス設計思想、生成ルール、DB 種別、決済手段、mock/preview 方法、本番設定、GitHub Actions 生成ルールまで記述。
- プロンプトは「Vibe プロジェクト標準仕様書」かつ最大の知財。これを起点に AI が対話フローを制御し、生成・検証・デプロイを一貫して行う。

## 体験の特徴
- ユーザとの対話はビジネス成果に絞り、専門用語を排除。「こういうアプリを作ります、よろしいですか？」に対して「よくわからないけど OK」で進行可能。
- preview デプロイ前に chrome-devtools-mcp で自動検証（lint / test / 主要フローのクリックチェック）。エラーがあればデプロイを止め、AI が再修正。
- 生成後はプレビューを確認してから、コード買い切り or SaaS 利用料で決済。コード所有権を得られる点が大きな価値。

## ビジネスモデル
- preview まで無料または低価格で提供し、満足度を高めた上で「コード買い切り」または「SaaS 利用料」で課金。
- コード所有権を明確化し、将来的な社内/外部エンジニアへの引き渡しや拡張を容易にする。
- 追加収益：カスタムプロンプトの業種別テンプレート販売、運用保守サポート、追加機能のオンデマンド生成。

## 将来拡張の方向性
- プロンプトパッケージのマーケットプレイス化（業種別標準仕様の流通）。
- 自動検証の深度拡大（E2E、パフォーマンステスト、自動セキュリティチェック）。
- マルチクラウドへの本番デプロイ最適化と、運用ダッシュボードによる健全性モニタリング。
